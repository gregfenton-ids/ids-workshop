# IDS AI Skeleton — Project Structure

**IDS AI Skeleton** is a modern, cloud-based *Dealership Management System* designed for RV and Marine dealerships. Built as an Nx monorepo with a NestJS backend and React frontend.

## Folder Layout

```
ids-ai-skeleton/
├── apps/
│   ├── astra-apis/          # NestJS backend
│   ├── astra-apis-e2e/      # Backend integration tests
│   ├── client-web/          # React frontend (SPA, SSR disabled)
│   │   └── app/
│   │       ├── components/  # Shared UI components
│   │       ├── core/        # Auth kernel, API client, middleware, hooks, storage
│   │       ├── icons/       # Custom SVG icons
│   │       ├── locales/     # i18n translations (en, fr)
│   │       ├── pages/       # Route page components
│   │       ├── stubs/       # Vite build stubs for backend-only packages
│   │       ├── types/       # TypeScript types
│   │       └── routes.tsx   # React Router route config
│   └── client-web-e2e/      # Playwright E2E tests
├── libs/shared/data-models/ # @ids/data-models — shared DTOs & types (intentional barrel export)
├── database/seeds/          # Seed scripts
├── docs/                    # Documentation and coding standards
├── .ai-workflow/            # AI context files
├── .ai-plan/                # Implementation plans
└── .github/agents/          # Custom Copilot agents
```

> **Codebase Navigation**: Always list `apps/astra-apis/src/` and `apps/client-web/app/` to discover the full backend and frontend module structure before making changes.

## DTO Layer Architecture

**Pattern**: Each NestJS module has a consistent DTO structure with **strict input/output separation**.

```
apps/astra-apis/src/{entity}/
├── dto/
│   ├── {entity}-base.dto.ts          # Input (BaseDto + validators) & Output (BaseResponseDto)
│   ├── {entity}-create.dto.ts        # Create-specific: CreateDto + CreateResponseDto
│   ├── {entity}-update.dto.ts        # Update-specific: UpdateDto + UpdateResponseDto
│   ├── {entity}-list.query.dto.ts    # List query + ListResponseDto
│   └── index.ts                      # Barrel export (if needed)
├── {entity}.mapper.ts                 # All entity → DTO conversions (toDto, toDtoList)
├── {entity}.controller.ts             # REST endpoints
├── {entity}.service.ts                # Business logic (uses mappers)
├── entities/
│   └── {entity}.entity.ts            # RavenDB entity (plain TypeScript type)
└── __test__/                         # Unit tests
```

**Key Rules:**
1. **Naming**: `{entity}-{operation}[.action].dto.ts`
   - `{operation}` = get, create, update, list, delete, search
   - `[.action]` = optional suffix (e.g., "query" for list and search queries)
2. **Input vs Output**: 
   - Input DTOs have validators (`@IsNotEmpty()`, `@IsString()`, etc.)
   - Output DTOs have NO validators
   - Both kept in same file for clarity (distinct classes)
3. **Mappers**: All entity-to-DTO conversions in `{entity}.mapper.ts`
   - Functions: function names should follow this convention `to{entity}{operation}Dto` for request, `to{entity}{operation}ResponseDto` for response
   - Handle null/undefined coercion
   - Import response DTO types only (not entities)
4. **Base Classes**:
   - `{Entity}BaseDto` = input with validators
   - `{Entity}BaseResponseDto` = output with core fields, no validators
   - `{Entity}CreateResponseDto` extends `IdsBaseResponseDto` (shared base with `locationId`)
   - `{Entity}ListResponseDto` extends `{Entity}BaseResponseDto`, adds audit fields. Applies to all entities except `location`

**Examples**: See `apps/astra-apis/src/{location,part,vendor,bin}/` for real implementations.

## Non-Obvious Architectural Facts

**Multi-tenancy**: The system is multi-tenant, and a tenant is an `ids_db` `Location` (e.g., `SMC`, `SRV`). Each `ids_db` `Location` is synchronized from a `logto_db` `Organization`. In the React app, users select their active `Location`; data and permitted functionality are scoped to that selected `Location`. At the data layer, tenant isolation is enforced via `location_id` on entities, so queries must always filter by `locationId`.

**Databases**: RavenDB for application data (`ids_db`). PostgreSQL for Logto auth (`logto_db`). Both defined in `docker-compose.yml`.

**Rendering**: Always check `apps/client-web/react-router.config.ts` for the current rendering mode (`ssr`, `prerender`). Before adding data fetching to a page, check existing pages for whether `loader` or `clientLoader` is used — match the established pattern.

**Shared library**: `@ids/data-models` (`libs/shared/data-models/`) is the single intentional barrel export. Import from `@ids/data-models` as normal.

**Auth**: Logto handles authentication (OAuth 2.0 / OIDC). Backend validates JWT via `@logto/node`, frontend uses `@logto/react`. Guards check permissions, location context flows from auth into queries.
